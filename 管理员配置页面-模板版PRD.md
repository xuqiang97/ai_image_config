# 业务背景和价值

## 业务背景

AI 生图能力已存在后端配置接口，可按用户、模型、appKey 和 quota 控制对应生图能力。此前因暂无前端资源，产品侧先通过 Postman 调用接口进行配置，能够满足临时使用，但不适合长期交接、日常维护和测试验收。

当前美工版和运营版的使用对象、配置类型、接口入参和底层数据边界不同，因此“管理员配置”需要在两版中分别建设独立页面，不能做成同一个页面里的美工 / 运营切换。

## 用户场景与痛点

| 人物角色 | 场景 | 痛点 | 解决方案 |
| --- | --- | --- | --- |
| 指定管理员 | 为指定用户配置某个生图模型的 appKey 和 quota | 通过 Postman 配置门槛高，字段和校验不直观 | 提供管理员配置页面，支持表单化新增和编辑 |
| 指定管理员 | 排查某用户某模型配置是否正确 | 缺少前端列表查询和详情查看入口 | 支持按用户ID、用户名、生图模型、appKey、启用状态查询 |
| 指定管理员 | 同一 appKey 下多条配置需要统一调整 quota | 逐条调整成本高，容易漏改 | 支持按 appKey 批量更新命中记录的 quota |
| 研发、测试、技术支持 | 联调、验收或排查接口行为 | 后端接口已上线但无单独接口文档 | 页面和文档明确以前端原型、README、群内 Postman 导出和已上线接口为准 |

## 业务价值

1. 降低管理员配置门槛，减少依赖 Postman 或人工协助调用接口。
2. 提升配置查询、编辑和批量调整效率，减少误配和漏配。
3. 固化美工版、运营版独立页面和数据边界，避免两版配置混用。
4. 明确 quota 校验和展示规则，降低前端、后端、测试之间的口径偏差。

本需求价值以效率提升、配置准确性提升和联调成本降低为主，暂不单独按金额量化。

# 功能需求

## 1. 系统结构和页面入口

在同一套公司系统账号下，左侧菜单区分美工版和运营版两个独立页面 / 路由，不提供同页切换：

| 页面 | 使用对象 | 页面入口 | 实现边界 |
| --- | --- | --- | --- |
| 美工管理员配置 | 设计部使用 | 新版 AI 生图 / 美工管理员配置 | 独立前端页面，沿用美工版已上线接口和数据边界 |
| 运营管理员配置 | 深圳运营、成都运营共用 | 新版 AI 生图 / 运营管理员配置 | 独立前端页面，沿用运营版已上线接口和数据边界 |

原型左侧“美工 / 运营”导航仅用于需求评审时快速对比，不是产品功能。实际交付时不提供同页切换器。

## 2. 两版差异

除下表外，美工版与运营版的功能能力、视觉和交互保持一致。

| 维度 | 美工版 | 运营版 |
| --- | --- | --- |
| 页面入口 | 美工管理员配置 | 运营管理员配置 |
| 页面路由 | 美工版独立页面路由 | 运营版独立页面路由 |
| 数据范围 | 美工场景下的 AI 生图配置 | 运营场景下的 AI 生图配置 |
| 接口入参 | 按已上线接口约定匹配“美工”配置类型 | 按已上线接口约定匹配“运营”配置类型 |
| 实现边界 | 沿用美工版已上线接口和数据边界 | 沿用运营版已上线接口和数据边界 |

两版 v1 都支持 LinkFox（1）、自研（2）、阿里绘蛙（3）；这些不再作为版本差异。

## 3. 主业务流程

1. 用户从左侧菜单进入美工版或运营版“管理员配置”页面。
2. 页面默认空筛选、第一页展示，列表默认按更新时间倒排。
3. 用户可通过用户ID、用户名、生图模型、appKey、启用状态查询配置。
4. 用户可新增配置，提交前完成必填、唯一性和 quota 校验。
5. 用户可查看单条配置详情，并可从查看进入编辑。
6. 用户可编辑 appKey、quota、启用状态；用户ID、生图模型、用户名只读。
7. 用户可删除单条配置，删除前需二次确认。
8. 用户可按 appKey 批量更新命中记录的 quota；未命中时保留抽屉输入，便于调整后重试。

## 4. 页面结构

页面自上而下为：

1. 顶部系统标题：AI生图管理员配置原型。
2. 左侧页面入口：美工管理员配置、运营管理员配置；仅用于原型评审对比。
3. 页面标题：美工管理员配置 / 运营管理员配置。
4. 页面提示：由指定管理员统一配置，其他人请勿动谢谢，排序默认按更新时间倒排。
5. 筛选区：用户ID、用户名、生图模型、appKey、启用状态、查询、重置。
6. 操作区：新增配置、按 appKey 批量更新。
7. 主内容：配置列表。
8. 分页器。
9. 右侧抽屉：查看、新增、编辑、批量更新。
10. 删除确认弹窗。

具体视觉和布局以页面原型为准。

## 9. quota 规则

quota 支持结构化填写和 JSON 直填，两种方式提交前使用同一套校验。接口返回的 JSON key 顺序不作为业务含义，前端展示时统一整理为：

`total -> scene -> closeUp -> sellPoint -> closeUpWhite -> aplus`

其他未列出的 key 排在上述固定 key 之后。

| 规则项 | 要求 |
| --- | --- |
| 类型 | 必须为 JSON 对象 |
| 空对象 | 不允许 |
| key | 仅允许字母、数字、下划线，且必须以字母或下划线开头 |
| key 重复 | 不允许 |
| value | 必须为非负整数，不限制上限 |

示例：

```json
{
  "total": 1000,
  "scene": 3,
  "closeUp": 2,
  "sellPoint": 2,
  "closeUpWhite": 0,
  "aplus": 7
}
```

补充说明：

1. `total` 按非负整数校验，不设置上限，例如 1000。
2. 允许 quota value 为 0。
3. 不允许负数、小数或非数字。
4. 前端展示顺序固定，但接口返回的 key 顺序不作为业务含义。
5. 后续若新增 quota 类型，前端结构化填写需支持新增 key；固定展示顺序可按产品确认后的 key 列表补充，未知 key 排在已知 key 之后。

## 10. 权限、接口与联调边界

本需求对应的后端接口已提前上线，暂无单独接口文档。配置类型取值、接口入参和权限 key 以群内 Postman 导出、已上线接口和后台权限体系为准。

| 边界项 | 要求 |
| --- | --- |
| 菜单 / 路由权限 | 控制用户是否能进入美工管理员配置、运营管理员配置页面 |
| 按钮权限 | 如后台已有细粒度控制，应覆盖新增、编辑、删除、批量更新 |
| 接口鉴权 | 以后端已上线鉴权为准；前端隐藏入口或按钮不能替代后端鉴权 |
| 无权限状态 | 按后台规范展示无权限提示或跳转无权限页面 |
| 接口成功 | 按操作刷新列表 |
| 接口失败 | 优先展示接口返回文案 |
| 防重复提交 | 查询、提交、删除、批量更新时应有加载态或按钮禁用态 |
| 空状态 | 列表无数据展示空状态，不应误报为接口失败 |

# 风险描述

| 风险 | 说明 | 应对策略 |
| --- | --- | --- |
| 两版页面被误做成同页切换 | 原型左侧有美工 / 运营导航，容易被误解为线上同页切换 | 明确两版必须是独立页面 / 路由；原型切换仅用于评审对比 |
| 接口字段或返回行为与页面需求不一致 | 后端接口已提前上线且暂无单独接口文档 | 联调以群内 Postman 导出和已上线接口为准；不一致时联系后端联调支持 |
| quota 类型后续扩展 | 当前固定展示顺序只列出已知 key | 支持未知 key 排在已知 key 后；新增类型后按产品口径补充固定展示顺序 |
| 权限只做前端隐藏 | 前端隐藏入口或按钮无法替代接口安全 | 接口权限以后端鉴权为准，前端仅做入口和按钮可见性控制 |
| 批量更新误操作 | 批量更新按 appKey 命中后会整包覆盖 quota | 抽屉顶部展示风险提示；可选生图模型缩小范围；未命中时保留输入便于调整 |

6. 新增配置时生图模型默认不选，用户ID、生图模型、用户名、appKey、quota 均必填。
7. 同一页面范围内，同一用户ID + 同一生图模型不能重复新增。
8. 查看详情展示 ID、用户ID、用户名、生图模型、appKey、启用状态、创建时间、更新时间和完整 quota JSON。
9. 编辑时用户ID、生图模型、用户名只读，appKey、quota、启用状态可编辑。
10. 删除前需二次确认，删除成功后刷新列表。
11. 批量更新按 appKey 命中，可选生图模型缩小范围，命中记录的 quota 被整包覆盖。
12. 批量更新未命中记录时，在 appKey 下方提示“未命中任何记录”，并保留抽屉输入。
13. quota 结构化填写和 JSON 直填规则一致；value 为非负整数且不限上限。
14. 前端展示 quota 时按 total、scene、closeUp、sellPoint、closeUpWhite、aplus 固定顺序整理。
15. 抽屉和删除弹窗支持关闭按钮与 Esc 关闭，关闭后尽量恢复到触发入口。
16. 美工管理员配置和运营管理员配置统一使用蓝色主题，按钮等可聚焦控件的键盘焦点态统一使用蓝色。
17. 查询、提交、删除、批量更新有加载态或按钮禁用态，避免重复提交。
18. 无权限、接口失败、列表无数据等状态处理符合后台规范和页面要求。

# 初始化用户权限

本需求涉及两个新增页面的菜单路由权限；如后台已有功能按钮权限控制，应同步覆盖新增、编辑、删除、批量更新。具体人员名单和权限 key 以前端所在后台权限体系配置为准。

正式使用权限和联调验收权限需区分配置：指定管理员用于线上日常维护；产品、研发、测试、技术支持如需参与联调、验收或排查问题，可按项目需要配置临时权限，验收完成后按后台权限管理规范回收或调整。

| 菜单路由 | 角色 | 授权范围 |
| --- | --- | --- |
| 美工管理员配置 | 指定管理员 | 授权进入美工管理员配置页面，管理美工场景下的 AI 生图配置 |
| 运营管理员配置 | 指定管理员 | 授权进入运营管理员配置页面，管理运营场景下的 AI 生图配置 |
| 美工管理员配置 / 运营管理员配置 | 本项目产品经理、本项目前后端研发、本项目测试、技术支持 | 按项目需要配置临时权限，用于联调、验收和问题排查 |

| 功能按钮 | 权限说明 |
| --- | --- |
| 新增配置 | 如后台已有按钮权限，需控制新增入口可见和接口可用 |
| 编辑 | 如后台已有按钮权限，需控制编辑入口可见和接口可用 |
| 删除 | 如后台已有按钮权限，需控制删除入口可见和接口可用 |
| 按 appKey 批量更新 | 如后台已有按钮权限，需控制批量更新入口可见和接口可用 |

前端隐藏入口或按钮不能替代后端接口鉴权。
